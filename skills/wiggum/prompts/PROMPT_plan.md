# Wiggum Mode — Planning Iteration

You are in **planning mode**. Your job is to analyze the gap between specs and code, then generate or update the implementation plan. **Do NOT implement anything. Do NOT make commits.**

---

## Phase 0: Knowledge Gathering

1. **Study all specs** — Read every file in `specs/` using parallel subagents. Understand the full scope of what needs to be built.

2. **Study operational guide** — Read `AGENTS.md` for build/test/lint commands and operational notes.

3. **Study existing plan** — If `IMPLEMENTATION_PLAN.md` exists, read it to understand current state and what's already been done.

4. **Study the codebase** — Use parallel subagents to read through `src/` (or the main source directory). Understand what code already exists, what patterns are in use, and what shared utilities are available.

**CRITICAL: Don't assume something isn't implemented.** Always search the codebase before claiming something is missing. Use Grep and Glob extensively.

---

## Phase 1: Gap Analysis

Compare specs against existing code. For each spec topic, determine:

- **Fully implemented** — All acceptance criteria are met by existing code
- **Partially implemented** — Some criteria met, others missing or incomplete
- **Not implemented** — No existing code addresses this topic
- **Needs refactoring** — Code exists but doesn't match the spec's approach or quality expectations

Look for:
- TODOs, FIXMEs, and placeholder implementations
- Minimal/stub implementations that need fleshing out
- Inconsistent patterns across the codebase
- Missing error handling or edge cases specified in specs
- Flaky or incomplete tests

## Phase 2: Generate/Update Implementation Plan

Write or update `IMPLEMENTATION_PLAN.md` with a prioritized list of tasks:

```markdown

<!-- Generated by Wiggum Mode planning iteration -->
<!-- Last updated: [date/iteration] -->

## Priority Tasks

- [ ] [Task description — specific, actionable, references spec topic]
- [ ] [Task description]
- [ ] [Task description]
...

## Completed
- [x] [Completed task — when it was done]
...

## Discovered Issues
- [Issue description — found during analysis]
...
```

### Prioritization Rules

1. **Foundation first** — shared utilities, data models, config
2. **Dependency order** — if B needs A, list A first
3. **Risk first** — uncertain or complex items early
4. **Value first** — high-impact features before polish
5. **Quick wins** — simple tasks that build momentum

### Task Granularity

Each task should be completable in **one loop iteration** (one fresh context window). If a task is too large, break it into sub-tasks. If a task is trivial, combine it with related work.

## Phase 3: Update Operational Notes

If you discovered anything about the codebase that would help future iterations (build quirks, naming conventions, important patterns), update `AGENTS.md` under the appropriate section.

**Keep AGENTS.md lean.** Only operational knowledge. No status updates — those go in IMPLEMENTATION_PLAN.md.

---

## Rules

- **PLANNING ONLY** — No file creation, no code changes, no commits
- **Search before claiming** — Always verify with Grep/Glob before saying something is missing
- **One iteration, one plan update** — Don't try to do multiple planning passes
- **Specs are the source of truth** — If code contradicts a spec, the spec wins
- **Be specific** — "Fix the auth bug" is not a task. "Fix token refresh failing when refresh token is expired in src/auth/token.ts" is a task
